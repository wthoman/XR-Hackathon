#### **GitHub Repository**

Visit our [specs-devs GitHub organization](https://github.com/specs-devs) for comprehensive access to all our development resources.

## Project Structure Best Practices

These are guidelines to help maintain clean, organized, and maintainable projects.

<CollapsibleSection title="Directory Organization" defaultOpen={false}>

Keep your contributions organized within a **`Project/`** folder in your Assets directory:

```
Assets/
├── Project/                    # Your main contribution folder
│   ├── Scripts/               # TypeScript/JavaScript source files
│   ├── Materials/             # Custom materials
│   ├── Shaders/               # Custom shader code
│   ├── Textures/              # Texture files
│   ├── Images/                # Image assets
│   ├── Prefabs/               # Reusable scene objects
│   ├── Animations/            # Animation files
│   ├── Sound/                 # Audio files
│   └── Data/                  # JSON configs, data files
├── Render/                    # Rendering-related components
│   ├── CameraTexture/
│   ├── EnvironmentMaps/
│   └── [camera layer configs]
└── Examples/                  # Standout examples (optional)
```

### Special Folders

**Render Folder:**

- Keep all render components together (Camera textures, Environment maps, etc.)
- Place anything dealing with camera layers and rendering here
- Keeps rendering concerns separate from game logic

**Examples Folder:**

- Used when you want users to discover something right away
- Ideal for standout examples that showcase key features

**Imported Assets:**

- For complex imports (e.g., Sketchfab models with many dependencies)
- Keep them in their own folder with all related files
- Acceptable to have these live independently if they contain animations, textures, and other bundled content

</CollapsibleSection>

<CollapsibleSection title="Language-Specific Organization">

### Dual Language Projects (TypeScript + JavaScript)

If your codebase exists in both JS and TS, **we recommend using the file suffix approach** for clarity:

```
Scripts/
├── MyComponentTS.ts           # TypeScript version
├── MyComponentJS.js           # JavaScript version
├── HelperTS.ts                # TypeScript helper
└── HelperJS.js                # JavaScript helper
```

This naming convention makes it immediately clear which language version you're working with and prevents confusion when both implementations coexist in the same directory.

</CollapsibleSection>

### Variable Hints

Add type hints and descriptive names:

<Tabs>
<TabItem value="TypeScript">

```typescript
// ✅ Good - Clear intent and type
private movementSpeed: number = 5.0
private targetPosition: vec3 = vec3.zero()
private isInteracting: boolean = false

// ❌ Bad - Unclear purpose
private speed = 5.0
private pos = vec3.zero()
private flag = false
```

</TabItem>
<TabItem value="JavaScript">

```js
// ✅ Good - Clear intent (use descriptive names)
/** @type {number} */
var movementSpeed = 5.0;
/** @type {vec3} */
var targetPosition = vec3.zero();
/** @type {boolean} */
var isInteracting = false;

// ❌ Bad - Unclear purpose
var speed = 5.0;
var pos = vec3.zero();
var flag = false;
```

</TabItem>
</Tabs>

### Lifecycle Methods

Always include lifecycle methods, even if empty:

<Tabs>
<TabItem value="TypeScript">

```typescript
@component
export class MyComponent extends BaseScriptComponent {
  onAwake() {
    if (this.enableLogs) {
      print('[MyComponent] onAwake called');
    }
    // Initialization logic

    this.createEvent('OnStartEvent').bind(() => {
      if (this.enableLogs) {
        print('[MyComponent] OnStartEvent triggered');
      }
      // Start logic
    });

    this.createEvent('UpdateEvent').bind(() => {
      // Update logic (can be empty with log)
    });
  }
}
```

</TabItem>
<TabItem value="JavaScript">

```js
// @input bool enableLogs {"default": true}

function onAwake() {
  if (script.enableLogs) {
    print('[MyComponent] onAwake called');
  }
  // Initialization logic

  script.createEvent('OnStartEvent').bind(function () {
    if (script.enableLogs) {
      print('[MyComponent] OnStartEvent triggered');
    }
    // Start logic
  });

  script.createEvent('UpdateEvent').bind(function () {
    // Update logic (can be empty with log)
  });
}

onAwake();
```

</TabItem>
</Tabs>

### Error Handling and Logging

**Allow undefined and handle errors gracefully:**

<Tabs>
<TabItem value="TypeScript">

```typescript
@input
private targetObject: SceneObject | undefined

onAwake() {
    if (!this.targetObject) {
        print("[ERROR] MyComponent: targetObject is not assigned!")
        return
    }
    // Proceed with logic
}
```

</TabItem>
<TabItem value="JavaScript">

```js
// @input SceneObject targetObject

function onAwake() {
  if (!script.targetObject) {
    print('[ERROR] MyComponent: targetObject is not assigned!');
    return;
  }
  // Proceed with logic
}

onAwake();
```

</TabItem>
</Tabs>

**Implement logging controls:**

<Tabs>
<TabItem value="TypeScript">

```typescript
@component
export class MyComponent extends BaseScriptComponent {
  @input
  private enableLogs: boolean = true;

  @input
  private verboseLogs: boolean = false;

  private log(message: string, verbose: boolean = false) {
    if (!this.enableLogs) return;
    if (verbose && !this.verboseLogs) return;
    print(`[MyComponent] ${message}`);
  }

  onAwake() {
    this.log('Component initialized');
    this.log('Detailed initialization info...', true); // Verbose log
  }
}
```

</TabItem>
<TabItem value="JavaScript">

```js
// @input bool enableLogs {"default": true}
// @input bool verboseLogs {"default": false}

function log(message, verbose) {
  if (!script.enableLogs) return;
  if (verbose && !script.verboseLogs) return;
  print('[MyComponent] ' + message);
}

function onAwake() {
  log('Component initialized');
  log('Detailed initialization info...', true); // Verbose log
}

onAwake();
```

</TabItem>
</Tabs>

### Code Quality

**Avoid `var` keyword:**

```typescript
// ✅ Good
const maxHealth = 100;
let currentHealth = 100;

// ❌ Avoid
var maxHealth = 100;
```

**Use code formatters:**

- Use tools like **Prettier** for consistent formatting
- Maintains even spacing and syntax
- Makes collaboration easier

- [ ] Update imports in modified components

- [ ] Use `git add <file>` instead of `git add .`

- [ ] Verify no credentials are in the repository

## Sensitive File Management

Protecting sensitive information in your projects is critical. **Never commit tokens, API keys, passwords, or other credentials to Git repositories.**

<CollapsibleSection title="Best Practices for Credentials" defaultOpen={false}>

### What to Protect

- API tokens and keys
- Service account credentials
- Database passwords
- OAuth secrets
- Private encryption keys

### Snap Cloud Credentials

When working with Snap Cloud, you'll need to import a project that contains your credentials. **These files must NOT be pushed to Git.**

```gitignore
# Add to your .gitignore
SnapCloudProject/
**/credentials.json
**/supabase-config.json
```

### Remote Service Gateway

When using the Remote Service Gateway, **do not commit scenes where credentials are visible in the Inspector.**

**Before committing:**

1. Temporarily remove credentials from Inspector fields
2. Commit your changes
3. Re-add credentials locally for testing

**Alternative approach:**

- Store credentials in a separate, gitignored configuration file
- Load them at runtime instead of hardcoding in the scene

</CollapsibleSection>

<CollapsibleSection title="General Security Tips">

### Keep Communication Private

Since there is no built-in cloud secret management system, keep communication about sensitive resources very private:

- Share credentials only through secure channels (encrypted messages, password managers)
- Never share credentials in public forums, Discord, or GitHub issues
- Rotate credentials immediately if they're accidentally exposed

### Environment-Specific Files

Create separate files for development and production:

```
Assets/
├── Config/
│   ├── development.json       # Local dev credentials (gitignored)
│   ├── production.json        # Prod credentials (gitignored)
│   └── config.template.json   # Template without actual values (committed)
```

</CollapsibleSection>

## Collaborations on Git

Git collaboration for Lens Studio projects requires special consideration due to how scenes are managed.

<CollapsibleSection title="Scene Management Strategy" defaultOpen={false}>

### Scene Master Approach

**Rule of thumb:** Scene management in Lens Studio differs from Unity. Until improvements are made in this area, follow these guidelines:

**Designate a Scene Master:**

- One team member should be responsible for final scene changes
- Other team members work on prefabs and assets
- Scene Master integrates prefabs into the main scene

**Why this approach?**

- Scene files can have complex merge conflicts
- Multiple people editing the same scene leads to lost work
- Scene hierarchy changes are difficult to merge automatically

### Workflow Example

```
Team Member A → Works on UI prefabs
Team Member B → Works on game logic scripts
Team Member C → Works on visual assets
Scene Master  → Integrates all prefabs into main scene
```

</CollapsibleSection>

<CollapsibleSection title="Prefab-Driven Development">

### Leverage Prefabs

Keep your work modular and reusable:

```
Assets/
├── Project/
│   ├── Prefabs/
│   │   ├── UI/              # UI components by Team Member A
│   │   ├── Gameplay/        # Game objects by Team Member B
│   │   └── Effects/         # VFX by Team Member C
│   └── Scripts/             # Scripts by all team members
```

### Benefits

- Team members can work independently on prefabs
- Prefabs are easier to merge than scene files
- Changes are isolated and testable
- Scene Master can easily integrate prefabs without conflicts

</CollapsibleSection>

<CollapsibleSection title="Smart Git Practices">

### Avoid `git add .` (Add All)

**Don't do this:**

```bash
git add .
git commit -m "Updated stuff"
```

**Do this instead:**

```bash
# Review what changed
git status

# Add specific files you edited
git add Assets/Project/Scripts/PlayerController.ts
git add Assets/Project/Prefabs/UI/MainMenu.prefab

# Verify what's staged
git diff --staged

# Commit with a clear message
git commit -m "Add player movement controls to PlayerController"
```

### Why Selective Commits Matter

- Prevents accidentally committing credentials
- Avoids committing temporary files or test assets
- Keeps commit history clean and reviewable
- Easier to revert specific changes if needed

### Clean Directory Organization

Separate work clearly across team members:

```
Assets/
├── Project/
│   ├── TeamMemberA/         # Alice's workspace
│   ├── TeamMemberB/         # Bob's workspace
│   ├── Shared/              # Shared resources
│   └── Integration/         # Scene Master's integration work
```

This structure minimizes conflicts and makes it clear who owns what.

</CollapsibleSection>
